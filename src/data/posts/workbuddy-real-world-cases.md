# WorkBuddy 实战：5 个真实工作场景的完整案例演示

上一篇讲了 WorkBuddy 是什么。这次来点硬核的——5 个真实工作场景，完整还原从下指令到拿成果的全过程。

每个案例都包含：**你给什么指令 → WorkBuddy 怎么执行 → 最终交付什么**。指令模板可以直接抄。

## 场景一：写周报

### 痛点
每周五下午最烦的事：回忆这周干了啥，凑字数，排版，发邮件。半小时起步。

### 我给的指令

```
帮我写本周周报，格式如下：

【本周完成】
1. 完成了用户登录模块的接口联调
2. 修复了购物车数量计算错误的 bug
3. 参加了周三的需求评审会

【下周计划】
1. 开始做支付模块
2. 优化首页加载速度

【需要协调】
支付模块需要后端先提供接口文档

要求：语言简洁，每条不超过 30 字，输出成 Word 文档。
```

### WorkBuddy 的执行过程
1. 解析我给的素材，分类归入"本周完成/下周计划/需要协调"
2. 精简每条描述，控制在 30 字以内
3. 调用文档生成工具，创建 `.docx` 文件
4. 输出文件供我下载

### 交付物
一份排版整齐的 Word 周报，打开就能用，改改就能发。

### 复用模板
```
帮我写周报。
【本周完成】[填你干的事，每件一行]
【下周计划】[填计划，每件一行]
【需要协调】[填需要别人配合的，没有就写"无"]
要求：[字数/格式/输出文件类型]
```

## 场景二：数据分析与可视化

### 痛点
老板丢来一份销售 Excel，要"看看有什么规律"。你要清洗数据、做透视表、画图表，至少一小时。

### 我给的指令

```
附件是一份 2025 年上半年销售数据（sales.xlsx），
包含日期、产品、地区、销售额四列。

请帮我分析：
1. 各产品的总销售额排名
2. 每月销售趋势
3. 哪个地区表现最好

分析完生成一张综合图表（柱状图+折线图组合），
导出成 PNG 图片，再附一份 300 字的结论摘要。
```

### WorkBuddy 的执行过程
1. 读取 Excel 文件，用 Pandas 加载数据
2. 检查数据质量（缺失值、异常值），必要时清洗
3. 按产品分组求和，排序
4. 按月份聚合，算趋势
5. 按地区分组对比
6. 用 Matplotlib 画组合图表，保存为 PNG
7. 根据数据写一段结论分析

### 交付物
- 一张 `sales_analysis.png` 图表
- 一段 300 字的文字结论，指出"产品 A 销额第一、3 月是峰值、华东地区领跑"

### 关键点
你不需要会 Pandas，不需要会 Matplotlib。你只需要会**提问题**——"排名、趋势、对比"就是三个最经典的分析维度。

## 场景三：行业调研报告

### 痛点
老板说"调研一下国内新能源汽车市场，明天给我一份报告"。你手动百度、开十几个标签页、复制粘贴，搞到大半夜。

### 我给的指令

```
帮我调研 2025 年中国新能源汽车市场，生成一份调研报告，
包含以下板块：

1. 市场规模与增长趋势
2. 主要品牌及市场份额
3. 消费者购买决策因素
4. 行业面临的挑战
5. 未来 3 年发展趋势预测

要求：
- 数据要有来源，标注引用
- 总字数 3000 字左右
- 输出成 PDF
- 附一份一页纸的摘要
```

### WorkBuddy 的执行过程
1. 联网搜索"2025 中国新能源汽车市场规模"等关键词
2. 抓取多个权威网页内容，提取数据
3. 交叉验证数据，标注来源
4. 按五大板块组织内容，撰写正文
5. 生成一页纸摘要
6. 调用 PDF 生成工具，导出 `.pdf`

### 交付物
- `new_energy_vehicle_report.pdf`（完整报告，约 3000 字）
- 一页纸摘要，直接发老板微信都行

### 经验
调研类任务，**一定要要求标注来源**。不然 AI 可能编数据。加了"数据要有来源，标注引用"这一句，WorkBuddy 会更谨慎地处理数据可信度。

## 场景四：快速搭建一个网站

### 痛点
想做个个人作品集网站，但不会前端，找外包又贵又慢。

### 我给的指令

```
帮我做一个个人作品集网站，要求：

- 单页应用，包含：个人介绍、项目展示、联系方式三个板块
- 现代简洁风格，主色调深蓝 + 白
- 响应式，手机端能用
- 项目展示用卡片布局，点击可看详情
- 顶部导航栏，滚动平滑跳转

我的信息：
- 姓名：尹聪朗
- 职业：全栈开发工程师
- 项目：[列了 3 个项目]

技术栈用 React，代码要规范，部署到 GitHub Pages。
```

### WorkBuddy 的执行过程
1. 初始化 React 项目（Vite 脚手架）
2. 创建组件：Navbar、Hero、About、Projects、Contact、Footer
3. 写 CSS 样式，深蓝白配色
4. 响应式适配（媒体查询）
5. 填入我提供的个人信息
6. 本地构建测试
7. 配置 GitHub Actions 自动部署
8. push 到仓库，触发部署

### 交付物
一个**上线的网站**，直接访问 URL 就能用。你现在看到的这个博客，就是类似流程做出来的。

### 核心价值
从"我想有个网站"到"网站上线"，全程不用写一行代码。你只需要描述"我想要什么样"。

## 场景五：批量处理邮件

### 痛点
出差一周回来，邮箱 200 多封。一封封看要死人，但又怕漏掉重要的。

### 我给的指令

```
帮我处理收件箱里的未读邮件（共 200+ 封），要求：

1. 按重要性分类：紧急 / 普通 / 可忽略
   - 老板和客户发的一律标记紧急
   - 系统通知、营销邮件标记可忽略
   - 其余为普通

2. 对紧急邮件，每封生成一段 50 字的摘要

3. 对普通邮件，汇总成列表（发件人 + 主题 + 一句话内容）

4. 可忽略的直接归档，不用看

5. 最后给我一份总览文档，按上面三类整理
```

### WorkBuddy 的执行过程
1. 连接邮箱（通过 IMAP 工具）
2. 拉取所有未读邮件的元数据和正文
3. 按规则分类
4. 对紧急邮件调用摘要模型
5. 对普通邮件提取关键信息
6. 可忽略邮件执行归档操作
7. 生成总览 Markdown 文档

### 交付物
一份 `inbox_overview.md`，你花 10 分钟扫一遍就知道这周错过了什么、哪些必须回。

## 五个场景的共同模式

看完这五个案例，你会发现一个规律。WorkBuddy 的指令都包含三个要素：

```
[背景/素材] + [具体要求] + [交付格式]
```

- 周报：本周事项 + 字数限制 + Word
- 数据分析：Excel 文件 + 三个分析维度 + PNG + 文字结论
- 调研报告：主题 + 五大板块 + PDF + 摘要
- 网站：需求描述 + 技术栈 + 上线
- 邮件：收件箱 + 分类规则 + 总览文档

**把这三要素写清楚，WorkBuddy 就能把活干漂亮。**
